# Private Database Connectivity (Task 4)

The backend connects to PostgreSQL entirely over the private network. The database
has **no public endpoint**. This document explains the design and how to verify it.

## What we built

PostgreSQL 14 runs on an Azure VM (`vm-db-devops-dev`) placed in a dedicated private
subnet (`snet-db`, `10.20.2.0/24`) with **no public IP**. The AKS nodes run in a
separate subnet (`snet-aks`, `10.20.1.0/24`) in the same VNet. A private DNS zone
(`devops.internal`) maps `postgres.devops.internal` to the VM's private IP
(`10.20.2.4`), resolvable only inside the VNet.

```
Internet ──HTTPS──> Azure LB ──> ingress-nginx ──> frontend ──/api──> backend
                                                                        │
                                          (private, 5432, NSG-restricted)│
                                                                        ▼
                                        postgres.devops.internal (10.20.2.4)
                                        PostgreSQL VM in snet-db, NO public IP
```

## How AKS connects privately to the database

The backend Pods run on nodes in `snet-aks`. Both subnets are in the same VNet, so
Pods reach the DB VM over private IP space — traffic never leaves the VNet and never
touches the public internet. The backend uses the hostname `postgres.devops.internal`,
which the VNet-linked private DNS zone resolves to `10.20.2.4`.

*(On AWS/EKS the equivalent is: EKS nodes in private subnets, an RDS instance in
private DB subnets within the same VPC, a Route 53 private hosted zone for the name,
and security groups in place of NSGs.)*

## Private subnet design

- `snet-aks` (`10.20.1.0/24`) — AKS nodes.
- `snet-db` (`10.20.2.0/24`) — the database VM only, no public IP.

Splitting them lets us apply a strict firewall to the DB subnet independent of the
cluster subnet. (When we briefly used the managed PostgreSQL Flexible Server, this
subnet was *delegated* to the service and paired with a Microsoft-managed private DNS
zone — the same private-access idea; we switched to a VM because the subscription's
offer blocked Flexible Server in the region.)

## Private DNS requirement

Applications should connect by name, not by a hardcoded IP. `azurerm_private_dns_zone`
`devops.internal` holds an A record `postgres → 10.20.2.4` and is linked to the VNet.
The name resolves **only inside the VNet** — it does not exist on the public internet.
This also means the DB can be rebuilt with a new IP and the backend config never
changes.

## NSG rules — how only the backend can access the database

The NSG on `snet-db` enforces least-privilege access to port 5432:

| Priority | Rule | Effect |
|---|---|---|
| 100 | Allow TCP 5432 from `10.20.1.0/24` (AKS subnet) → DB subnet | Only the cluster can connect |
| 200 | Deny TCP 5432 from `VirtualNetwork` → DB subnet | Nothing else in the VNet can |

Combined with the VM having no public IP, the database is reachable **only** from the
AKS subnet on the database port. Anything else — other subnets, the internet — is
blocked.

## How database credentials are stored securely

- The DB password is **generated by Terraform** (`random_password`) — never written in
  code. It lives only in the remote state (in a locked-down, non-public storage account).
- It is injected into Kubernetes as a **Secret**, created at deploy time straight from
  the Terraform output, so it never touches git:
  ```bash
  kubectl -n app create secret generic backend-secret \
    --from-literal=DB_PASSWORD="$(terraform -chdir=terraform output -raw db_admin_password)"
  ```
- The backend reads it as an environment variable (`DB_PASSWORD`) from that Secret;
  non-secret settings (host, port, db, user) come from a ConfigMap.
- The repo only contains `backend-secret-example.yaml` with a placeholder value.
- **Production-grade next step:** back the Kubernetes Secret with **Azure Key Vault** via
  the Key Vault CSI driver, so the password is never stored in etcd in plaintext.

## How to confirm the database is NOT publicly accessible

1. **No public IP on the VM:**
   ```bash
   az vm show -g rg-devops-dev -n vm-db-devops-dev -d \
     --query "{privateIps:privateIps, publicIps:publicIps}" -o table
   ```
   `publicIps` is empty.

2. **The name does not resolve publicly** — `postgres.devops.internal` only resolves
   inside the VNet (from a Pod), not from the public internet.

3. **Reachable from inside the AKS subnet, and only there:**
   ```bash
   kubectl -n app run nettest --rm -it --image=busybox --restart=Never -- \
     sh -c 'nc -zv -w 5 postgres.devops.internal 5432'   # -> 10.20.2.4:5432 open
   ```

4. **The app proves a real query end-to-end:**
   ```bash
   curl -s https://app.chishty.me/api/db
   # {"connected":true,"host":"postgres.devops.internal","database":"appdb", ... }
   ```
